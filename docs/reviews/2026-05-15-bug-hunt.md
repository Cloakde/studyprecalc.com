## V1 — Invite consumed before email verification — IMPORTANT [verified]

**Where:** `supabase/schema.sql` — `handle_new_user()` (`after insert on auth.users`), the `update public.invites set consumed_at ...` block around lines 341-345.

**Bug:** The invite is marked `consumed_at = now(), consumed_by = new.id` the instant the `auth.users` row is created (during `supabase.auth.signUp()`), which is **before** the AUTH-008 email code is verified. An abandoned or never-confirmed signup permanently burns a one-time invite with no reclaim path.

**Fix:** Keep validation + profile creation at insert, but move invite **consumption** and class enrollment to a new trigger that fires when `email_confirmed_at` transitions `NULL → timestamp`.

1. In `handle_new_user()`, delete the invite-consumption `update` and the class-enrollment `insert` (the invite *validation* `select … raise exception` checks stay — bad invites must still reject the signup early).

2. Add a confirmation trigger:

```sql
create or replace function public.handle_user_email_confirmed()
returns trigger
language plpgsql
security definer
set search_path = public
as $$
declare
  invite_code text := public.normalize_invite_code(new.raw_user_meta_data ->> 'invite_code');
  invite_record public.invites%rowtype;
  normalized_email text := lower(trim(coalesce(new.email, '')));
  display_name text := coalesce(new.raw_user_meta_data ->> 'display_name',
                                split_part(coalesce(new.email, ''), '@', 1), 'Student');
begin
  -- only on the NULL -> confirmed transition
  if old.email_confirmed_at is not null or new.email_confirmed_at is null then
    return new;
  end if;
  if invite_code = '' then
    return new;
  end if;

  select * into invite_record from public.invites
  where code = invite_code
  for update;

  -- invite already gone (revoked, expired, or claimed by another confirmed user):
  -- the profile already exists from the insert trigger; just stop.
  if not found
     or invite_record.consumed_at is not null
     or invite_record.revoked_at is not null
     or (invite_record.expires_at is not null and invite_record.expires_at < now()) then
    return new;
  end if;

  update public.invites
  set consumed_at = now(), consumed_by = new.id
  where id = invite_record.id;

  if invite_record.class_id is not null then
    insert into public.class_enrollments (class_id, account_id, email, display_name, role)
    values (invite_record.class_id, new.id, normalized_email, display_name, invite_record.role)
    on conflict (class_id, account_id) do update
      set email = excluded.email, display_name = excluded.display_name;
  end if;

  return new;
end;
$$;

drop trigger if exists on_auth_user_email_confirmed on auth.users;
create trigger on_auth_user_email_confirmed
after update of email_confirmed_at on auth.users
for each row execute function public.handle_user_email_confirmed();
```

**Trade-off to document, not block on:** between insert and confirmation the invite is not held, so two unconfirmed signups could both pass the insert-time check; only the first to confirm consumes it. The second confirms with a default `student` profile and no class enrollment — acceptable for an invite-only beta. If stricter behavior is needed later, add `reserved_at`/`reserved_by` columns, reserve at insert, treat reserved-but-unconsumed as unavailable, and reclaim on a TTL. Out of scope for the immediate fix.

## V2 — Email verification bypassable via Supabase project config — IMPORTANT [verified]

**Where:** `src/data/supabase/accountStore.ts:295-313` (`signup`).

**Bug:** If the Supabase project has "Confirm email" disabled (or auto-confirm on), `signUp` returns a live `data.session`, the code sets the current account immediately, and `requiresEmailConfirmation` is `false` — so the AUTH-008 verification screen never renders. Email verification is not enforced by the app; it silently depends on a dashboard toggle no code or runbook checks.

**Fix (defense in depth — apply both):**

1. Gate "authenticated" on a verified signal, not merely a returned session. In `signup`, replace the `if (data.session)` block:

```ts
const account = await loadProfileAccount(data.user);
const emailConfirmed = Boolean(data.user.email_confirmed_at);

if (data.session && emailConfirmed) {
  setCurrentAccount(account);
}

return {
  account,
  requiresEmailConfirmation: !emailConfirmed,
  ...(!emailConfirmed ? { verificationEmail: signupInput.email } : {}),
};
```

   Also audit `refreshCurrentAccount` / `onAuthStateChange` (same file, ~262-268, ~400-411): when a session exists but `session.user.email_confirmed_at` is null, do **not** `setCurrentAccount(account)` — route to the verification screen instead. Otherwise a reload re-authenticates an unverified user around the signup guard.

2. Add a hard requirement to the M8 owner runbook (`docs/operations/*`): **Supabase → Authentication → Email → "Confirm email" must be ON**, listed as required activation evidence (a screenshot or the setting value) before go-live.

## V3 — Logout leaves the session intact on `signOut` failure — IMPORTANT [verified]

**Where:** `src/data/supabase/accountStore.ts:378-389` (`logout`).

**Bug:** On `signOut` failure the function sets `lastError` and **returns early** without clearing `currentAccount` or the persisted `sb-*` token. A reload silently re-authenticates. The "clear local storage on failure" behavior the REVIEW-001A reply claimed was never implemented.

**Fix:** Always drop local auth state, even when the network sign-out fails. `supabase.auth.signOut({ scope: 'local' })` clears the persisted session without a network round-trip and is the correct fallback.

```ts
const logout = useCallback(async () => {
  setLastError('');
  const result = await signOutSupabaseAccount();

  if (!result.signedOut) {
    // Network/global sign-out failed — still terminate the local session
    // so a reload cannot silently re-authenticate.
    try {
      await supabase?.auth.signOut({ scope: 'local' });
    } catch {
      /* best effort: fall through to clearing React state regardless */
    }
    setCurrentAccount(null);
    setLastError(result.errorMessage);
    return;
  }

  setCurrentAccount(null);
  setLastError(result.errorMessage);
}, []);
```

If `signOutSupabaseAccount()` already attempts a local-scope sign-out internally, skip the inner call and just keep the unconditional `setCurrentAccount(null)` — the essential change is that the early-return path must not leave `currentAccount` populated.

## V4 — `SessionPractice.submitSession` has no phase guard — IMPORTANT [verified]

**Where:** `src/app/components/SessionPractice.tsx:543-551`.

**Bug:** `submitSession` does not early-return on `phase !== 'running'`. Timer auto-submit (`submitSessionRef.current()`, ~line 358) and the manual button (`onClick={submitSession}`, ~line 1152) can both fire — a click in the same tick as expiry, or a double click — running `saveSessionAttempts` + `saveGroupedSessionResult` twice and double-saving attempts/session results.

**Fix:** One line, plus add `phase` to the dependency array. This is the smallest, highest-value fix in this doc — land it independently and immediately.

```ts
const submitSession = useCallback(() => {
  if (phase !== 'running') {
    return;
  }
  const submittedAt = new Date();
  const nextResponses = saveSessionAttempts(submittedAt);
  saveGroupedSessionResult(nextResponses, submittedAt);
  setSessionSubmittedAt(submittedAt);
  setClockNow(submittedAt.getTime());
  setSummaryReviewFilter('all');
  setPhase('summary');
}, [phase, saveGroupedSessionResult, saveSessionAttempts]);
```

The ref-sync effect at `:553-555` (`[submitSession]`) keeps `submitSessionRef.current` pointing at the guarded version automatically — no other change needed.

## A1 — Content delete failure has no rollback — IMPORTANT [agent, code read]

**Where:** `src/data/supabase/questionContentStore.ts:690-705` (`deleteCustomQuestion`).

**Bug:** The record is optimistically removed from state (lines 693-695) before the async delete is enqueued. The `.catch` (line 703) only sets `contentError` — it never restores the removed record. A failed network delete makes the question vanish from the UI while it still exists on the server until a manual `refreshContent`.

**Fix:** Snapshot the record before optimistic removal; restore it on failure if the operation is still the latest for that id.

```ts
const deleteCustomQuestion = useCallback(
  (questionId: string) => {
    const operationVersion = nextQuestionOperationVersion(questionId);
    const removedRecord = contentRecordsRef.current.find(
      (record) => record.id === questionId,
    );
    setSortedContentRecords(
      contentRecordsRef.current.filter((record) => record.id !== questionId),
    );

    void enqueueQuestionWrite(questionId, async () => {
      await store.deleteQuestion(questionId);
      if (isLatestQuestionOperation(questionId, operationVersion)) {
        setContentError('');
      }
    }).catch((error) => {
      if (isLatestQuestionOperation(questionId, operationVersion)) {
        // Roll back the optimistic removal so UI and server agree.
        if (removedRecord) {
          setSortedContentRecords([
            ...contentRecordsRef.current.filter((r) => r.id !== questionId),
            removedRecord,
          ]);
        }
        setContentError(
          error instanceof Error ? error.message : 'Unable to delete question content.',
        );
      }
    });
  },
  [/* unchanged deps */],
);
```

Apply the same snapshot-and-restore pattern to `setCustomQuestionStatus` if it optimistically mutates before the network call.

## A2 — `syncQuestionMediaLinks` nullable-conflict + non-atomic — IMPORTANT [agent, code read] — needs a migration

**Where:** `src/data/supabase/questionContentStore.ts:285-329`, plus the `question_media` unique index in `supabase/schema.sql`.

**Bug:** `upsert(..., { onConflict: 'question_id,placement,asset_id' })` with a **nullable `asset_id`**. In Postgres a unique index treats `NULL` values as distinct unless declared `NULLS NOT DISTINCT` (PG15+). So link rows where `asset_id IS NULL` never conflict-match → the upsert inserts duplicates instead of updating. Separately, the `questions` row is persisted earlier in `persistRecord` and this link sync runs afterward with no transaction — a failure here leaves the question saved but media links stale/duplicated with no rollback.

**Fix (two parts — schema migration + app):**

1. Make the conflict target deterministic for null asset ids. Preferred: store a non-null sentinel instead of `NULL` for "question-level, no asset" links (e.g. empty string), and keep the existing 3-column unique index. Migration:

```sql
-- one-time backfill + constraint
update public.question_media set asset_id = '' where asset_id is null;
alter table public.question_media alter column asset_id set default '';
alter table public.question_media alter column asset_id set not null;
-- ensure the unique index matches the upsert onConflict target
create unique index if not exists question_media_qpa_key
  on public.question_media (question_id, placement, asset_id);
```

   (Alternative if you must keep NULLs: `create unique index ... (question_id, placement, asset_id) nulls not distinct;` on PG15+, and confirm the Supabase Postgres version supports it.)

   Then in the app, map `asset.assetId ?? ''` when building both `desiredLinkKeys` and the upsert rows so the key space matches the DB exactly (line 295 already uses `?? ''` for the stale check — make the upsert payload consistent at line 306: `asset_id: asset.assetId ?? ''`).

2. Make question-row + media-link writes atomic. Wrap the `questions` upsert and the `question_media` upsert/delete in a single `rpc` (a `security definer` SQL function taking the row + link array as JSON), so a link-sync failure rolls the whole save back. If an RPC is too large for this pass, the minimum stopgap is: on `syncQuestionMediaLinks` throw, surface a clear `contentError` **and** re-`refreshContent()` so the UI reflects the half-written server state instead of the optimistic success.

This is the one finding that needs a schema migration and a careful read of the live Supabase PG version — do not land it as a blind one-liner.

## A3 — `revokeInvite` shows success on a silent no-op — IMPORTANT [agent, code read]

**Where:** `src/data/supabase/inviteStore.ts:526-561`.

**Bug:** `.update({ revoked_at }).eq('id', inviteId).select('*').maybeSingle()` returns `data: null` with **no error** when the row doesn't exist or RLS silently blocks the update. The code clears `lastError` and falls through to the local `markInviteRevoked` branch (554-558), optimistically showing the invite as revoked even though the server row was never updated. A still-valid invite looks revoked to the admin.

**Fix:** In the cloud path, treat `!data && !error` as a failure and return before the local fall-through.

```ts
if (enabled && supabase) {
  const { data, error } = await supabase
    .from('invites')
    .update({ revoked_at: revokedAt })
    .eq('id', inviteId)
    .select('*')
    .maybeSingle<InviteRow>();

  if (error) {
    setLastError(error.message);
    return;
  }

  if (!data) {
    setLastError('Invite could not be revoked — it no longer exists or you lack permission.');
    return;
  }

  setLastError('');
  const revokedInvite = inviteFromSupabaseRow(data);
  setInvites((currentInvites) =>
    currentInvites.map((invite) => (invite.id === inviteId ? revokedInvite : invite)),
  );
  return;
}

// local-only fallback (no Supabase configured)
setInvites((currentInvites) =>
  currentInvites.map((invite) =>
    invite.id === inviteId ? markInviteRevoked(invite, { revokedAt }) : invite,
  ),
);
```

## A4 — Zero/empty-rubric FRQs scored as correct — IMPORTANT [agent, code read]

**Where:** `src/domain/sessions/createSessionResult.ts:95` and `src/domain/attempts/createAttempt.ts:~93`. Root cause is `scoreFrqChecklist` returning `maxScore: 0` for an empty/all-zero rubric (`scoreFrq.ts:3-7`).

**Bug:** `isCorrect: score.score === score.maxScore` → `0 === 0` → `true` for an answered FRQ with no rubric points. Such questions are excluded from `missedQuestionIds` (`0 < 0` is false) and counted as correct, inflating dashboard accuracy.

**Fix:** A zero-max question is not "correct" — guard on `maxScore > 0`.

In `createSessionResult.ts:95`:

```ts
isCorrect: score.maxScore > 0 && score.score === score.maxScore,
```

In `createAttempt.ts` (`createFrqAttempt`, the `isCorrect` line ~93):

```ts
isCorrect: score.isCorrect ?? (score.maxScore > 0 && score.score === score.maxScore),
```

Defense in depth: add a content-validation rule in `scripts/validate-content.ts` rejecting an FRQ whose rubric totals 0 points (an FRQ with nothing to score is an authoring error). This stops the bad data at the door rather than only fixing the rollup math.
